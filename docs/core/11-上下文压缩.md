# 上下文压缩

Agent 在执行复杂任务时会进行多轮 LLM + Tool 调用，每轮都会往内存中的消息列表追加消息。以一个复杂任务为例，15-20 轮工具调用后，上下文中会累积大量的 ToolResponse（文件内容、命令输出）和 ToolCall 参数（完整代码脚本），这些内容很快就会逼近模型的上下文窗口限制。

这个问题的本质是：**Agent 的对话历史会持续膨胀，而模型的上下文窗口是有限的**。膨胀会导致：

- **成本上升**：每轮 LLM 调用都携带完整历史，Token 消耗递增
- **超出上下文窗口**：超出模型最大 Token 限制导致调用失败
- **性能下降**：过多的历史信息分散模型注意力，降低回答质量

框架提供两层自动压缩策略来应对不同级别的上下文膨胀，在每轮 LLM 调用前按需执行。

> 上下文压缩仅压缩**当前执行期间**内存中的消息列表，不影响持久化的会话记忆。

## 压缩流程

```
每轮 Agent Loop 调用 LLM 前：
┌──────────────────────────────────────────────────────┐
│ Layer 1: micro_compact（每轮自动执行）                  │
│                                                      │
│ 替换旧 ToolResponse 内容和 ToolCall 长参数 → JSON 占位符│
│ - 跳过 protectedTools 中的工具响应                      │
│ - ToolResponse 和 ToolCall 统一使用 maxToolLength     │
│ - 保留最近 keepRecentTools 轮的完整内容                 │
│ - 不调 LLM，纯字符串操作，开销极低                       │
└──────────────────────────────────────────────────────┘
        │
        ▼
┌──────────────────────────────────────────────────────┐
│ 检查: estimateTokens(messages) > threshold?           │
│   否 → 继续 LLM 调用                                   │
│   是 ↓                                                │
│ Layer 2: auto_compact（超阈值触发）                      │
│                                                      │
│ 1. 保留 SystemMessage（系统提示词）                      │
│ 2. 传入当前用户请求，引导摘要重点                         │
│ 3. 对所有其他旧消息调用 LLM 生成结构化摘要               │
│ 4. 重建：SystemMessage → 摘要 UserMessage              │
└──────────────────────────────────────────────────────┘
```

大部分情况下 Layer 1 就够了，Layer 2 只在极端情况下才触发。就好比 JVM 的 GC 策略：Young GC 频繁执行、开销低；Full GC 不常触发，但一旦触发就是彻底的清理。

## Layer 1：micro_compact（每轮自动执行）

micro_compact 是第一道防线，每轮 Agent Loop 自动执行，**不调用 LLM，纯字符串操作，开销极低**。

核心思路是"保留最近、压缩历史"：最近 N 轮的工具调用内容保持完整，更早的替换为 JSON 占位符。Agent 已经不需要 Round 1 那份 3000 字符的文件内容了，但它们仍然占据着上下文空间。

### 旧 ToolResponse 替换

超出 `keepRecentTools` 范围且内容超过 `maxToolLength` 的旧工具响应替换为 JSON 占位符：

```
压缩前：
  [ToolResponse] read_file → "# Spring AI AgentX\n基于 Spring AI 的智能体开发框架..."（3000 字符）

压缩后：
  [ToolResponse] read_file → {"compacted":true,"tool":"read_file","originalLength":3000,"message":"content compressed"}
```

### 旧 ToolCall 参数压缩

超出 `keepRecentTools` 范围且参数超过 `maxToolLength` 的旧 AssistantMessage 中的 ToolCall 参数替换为 JSON 占位符：

```
压缩前：
  arguments: "这是一段非常长的代码内容，包含5000个字符..."

压缩后：
  arguments: {"compacted":true,"tool":"write_file","originalLength":5000,"message":"args compressed"}
```

### 占位符为什么必须是 JSON

这看似简单的占位符格式，其实是踩过坑之后的改进。

一开始 ToolResponse 的占位符用的是人类可读的格式：`[Previous: used read_file, result was 3200 chars]`。但实际跑起来发现有些模型在处理对话历史时会尝试解析 ToolResponse 的内容，当看到一段方括号包裹的自然语言描述时，模型会"困惑"——轻则影响推理质量，重则报错。

而且 ToolCall 参数那边一开始就是 JSON 占位符，两种格式混在一起不统一。最终的解决方案：**ToolResponse 和 ToolCall 统一为 JSON 格式，字段命名保持一致**（`compacted`、`tool`、`originalLength`、`message`）。

> **原则**：在 LLM 的上下文中，工具返回的数据应该保持结构化格式（JSON），不要用自然语言描述来代替。

### 保护工具

受保护的工具的响应和参数不会被压缩。SkillsTool 是内置保护的，因为技能指令在后续每轮决策中都需要参考，压缩后 Agent 就"失忆"了。你也可以通过 `protectedTools()` 添加自定义保护工具。

### micro_compact 的效果

假设 Agent 执行了 10 轮工具调用，每轮 ToolResponse 平均 2000 字符、ToolCall args 平均 2000 字符。在 `keepRecentTools=4` 的情况下：

- 压缩前：10 轮 × 4000 = 40000 字符
- 压缩后：6 轮 × 160（JSON 占位符）+ 4 轮 × 4000 = 16960 字符
- **节省约 57% 的上下文空间**

但 micro_compact 有一个根本性的局限：它只是把"大内容"替换成"小占位符"，**消息的数量并没有减少**。当占位符累积到一定程度时，就需要 Layer 2 出场了。

## Layer 2：auto_compact（Token 超阈值时触发）

当 micro_compact 后 Token 仍超过阈值时，用 LLM 生成结构化摘要替换所有旧消息。

auto_compact 的思路非常直接：**不挑挑拣拣，把所有旧消息（除 SystemMessage）统统交给 LLM 生成一份结构化摘要，然后用这一条摘要替换掉全部旧消息**。

```
压缩前（20+ 条消息）：                              压缩后（2 条消息）：

[0]  SystemMessage（系统提示词）                    [0] SystemMessage（不变）
[1]  UserMessage（用户提问）                        [1] UserMessage（结构化摘要）
[2]  AssistantMessage（助手回复）
[3]  ToolResponseMessage（read_file 返回）
[4]  AssistantMessage（助手回复）
[5]  ToolResponseMessage（write_file 返回）
...
[20] AssistantMessage（最新回复）
```

从 20 多条消息直接变成 2 条：SystemMessage + 摘要。

### 为什么不做"精确切割"

之前的版本尝试过"精确切割"：保留最近几条消息不动，只压缩更早的消息。但这种方案有两个问题：

**代码复杂度**。Spring AI 中 ToolResponseMessage 不能独立存在，必须紧跟在包含对应 ToolCall 的 AssistantMessage 后面。如果切割点落在 ToolResponseMessage 上，前面没有对应的 AssistantMessage，Spring AI 直接报错。需要做"边界对齐"、"受保护消息提取"等处理，代码很快就变得非常复杂。

**压缩质量**。"半压缩半保留"的方式本身就有问题。保留的最近几条消息是脱离了上下文的——前面的对话被压缩成摘要，后面是原始消息。Agent 阅读这种"断层"上下文时可能产生理解偏差。

最终的解决方案是"全部摘要"：**所有消息都交给 LLM 处理**，在摘要 Prompt 中描述清楚应该保留哪些关键信息，至于哪些重要、哪些可以丢弃，交给摘要 LLM 来判断。代码极其简单，而且摘要 LLM 比任何硬编码规则都更灵活。

### 摘要知道"当前在干什么"

摘要请求发送给 LLM 时，会传入当前用户的问题（query）。这样 LLM 就知道应该重点关注什么——围绕当前任务保留关键信息，对已经完成或无关的部分做更激进的压缩。

> query 从 `call(query, params)` 入口透传到 `compact(messages, query)`，不需要从消息列表中反查。

### 结构化摘要提示词

auto_compact 的效果很大程度上取决于摘要 Prompt 的质量。摘要的核心目标是**任务延续性**——不是给人类看的总结报告，而是给"失忆后的 Agent"看的"备忘录"，Agent 读到摘要后必须能准确理解：

- 用户想要什么
- 已经做了什么
- 下一步该做什么

摘要 Prompt 要求保留的关键信息包括：

- **用户意图**：原始请求的完整含义，不能丢失任何细节
- **精确引用**：文件路径、URL、变量名原样保留，不能模糊化
- **工具调用结论**：每个工具调用产生了什么结果，成功还是失败
- **下一步操作**：明确接下来要调用什么工具、传什么参数

### 回退策略

如果 LLM 摘要生成失败（网络超时、API 限流等），自动回退到截断策略：保留最近 10 条消息的文本内容作为摘要。虽然效果远不如 LLM 摘要，但至少保证了不会导致灾难性的信息丢失。

## 使用方式

```java
// 方式 1：默认配置（阈值 60000, 保留最近 4 轮工具调用）
ReactAgent agent = ReactAgent.builder()
        .chatModel(chatModel)
        .contextPolicy(ContextPolicy.defaults())
        .build();

// 方式 2：自定义配置
ReactAgent agent = ReactAgent.builder()
        .chatModel(chatModel)
        .contextPolicy(ContextPolicy.builder()
                .tokenThreshold(15000)           // Token 阈值，超过触发 auto_compact
                .keepRecentTools(2)              // 保留最近 2 轮工具调用的完整内容
                .maxToolLength(300)              // 内容超过 300 字符时压缩
                .protectedTools("my_tool")       // 自定义保护工具（SkillsTool 已内置，无需手动添加）
                .build())
        .build();

// 方式 3：不配置 → 不启用压缩（默认行为，完全向后兼容）
ReactAgent agent = ReactAgent.builder()
        .chatModel(chatModel)
        .build();
```

## 配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `tokenThreshold` | 60000 | Token 估算阈值，超过时触发 auto_compact |
| `keepRecentTools` | 4 | micro_compact 保留最近 N 轮工具调用的完整内容 |
| `maxToolLength` | 200 | ToolResponse 内容和 ToolCall 参数的统一压缩阈值 |
| `protectedTools` | 自动包含 SkillsTool | 受保护的工具，响应和参数不会被压缩 |

### Token 阈值怎么设置

`tokenThreshold` 默认 60000，大约是主流模型 128K 上下文窗口的一半。不能太小也不能太大：

- **太小（如 20000）**：5-6 轮工具调用就可能触发 auto_compact，频繁触发意味着频繁调 LLM 生成摘要，增加延迟和成本，且每次压缩都会损失精度
- **太大（如 100000）**：auto_compact 基本不触发，下一轮 LLM 调用可能直接超出模型上下文窗口

> 宁可早一点触发，也不要等到上下文快爆了才处理。

### 参数调优建议

| 场景 | tokenThreshold | keepRecentTools | maxToolLength | 说明 |
|------|---------------|-----------------|---------------|------|
| 简单任务 | 不启用 | — | — | 轮次少，不需要压缩 |
| 中等任务 | 60000 | 4 | 200 | 默认配置即可 |
| 长任务（多工具调用） | 30000 | 2 | 150 | 更积极的压缩 |
| 极长任务（100+ 轮） | 15000 | 1 | 100 | 最激进的压缩 |

## Token 估算

框架使用内置的 `TokenEstimator` 进行 Token 估算，不依赖外部分词器：

- **英文/ASCII**：约 4 字符 = 1 Token
- **中文/CJK**：约 1.5 字符 = 1 Token

为什么中英文要分开算？因为分词器处理中英文的方式差异很大：英文 "hello world" 约 2 个 token（4 字符/token），而中文 "你好世界" 约 4 个 token（1-1.5 字符/token）。如果简单用"字符数 / 4"来估算中文，会严重低估 Token 消耗。

估算结果用于阈值判断，不需要精确。实际 Token 消耗由模型决定。
