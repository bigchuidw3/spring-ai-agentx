# 思考模型适配

部分模型支持"思考模式"，会在输出中返回推理过程（思考内容）和正式回答。不同厂商的思考模型通过不同方式返回推理内容，框架通过 `ThinkingMode` 枚举统一适配。

## 两种输出格式

| 格式 | 说明 | 适用模型 |
|------|------|---------|
| `THINK_TAG` | 思考内容嵌入在 `content` 字段中，用 `<think/>` 标签包裹 | MiniMax M2.7 等 |
| `REASONING_CONTENT` | 思考内容通过独立的 `reasoning_content` 字段返回，`content` 只有正文 | DeepSeek、Qwen3.6-plus 等 |

## 配置方式

通过 `thinkingMode()` 指定模型的思考内容输出格式：

```java
// DeepSeek / Qwen3.6-plus — reasoning_content 格式
ReactAgent agent = ReactAgent.builder()
        .chatModel(chatModel)
        .thinkingMode(ThinkingMode.REASONING_CONTENT)
        .build();

// MiniMax 等 — <think/> 标签格式
ReactAgent agent = ReactAgent.builder()
        .chatModel(chatModel)
        .thinkingMode(ThinkingMode.THINK_TAG)
        .build();

// 普通模型（默认，不处理思考内容）
ReactAgent agent = ReactAgent.builder()
        .chatModel(chatModel)
        .build();
```

> **入参配置由调用方负责**：不同模型开启思考的入参不同（如 Qwen 的 `thinking_enable`、DeepSeek 的 `thinking.type`），调用方在创建 `ChatModel` 时自行配置。框架只负责解析输出。

> **REASONING_CONTENT 模式建议统一使用 `OpenAiChatModel`**：Spring AI 各厂商实现类对 `reasoning_content` 的映射程度不一（如 DeepSeek 官方实现已映射，但 ZhiPu 原生实现未映射）。为确保兼容性，建议所有 `REASONING_CONTENT` 模式的模型统一通过 OpenAI 兼容接口接入。各大厂商均提供 OpenAI 兼容接口，配置时通过 `completionsPath` 调整路径即可：
> ```java
> // GLM 通过 OpenAI 兼容接口访问（支持 reasoning_content）
> ChatModel chatModel = OpenAiChatModel.builder()
>         .openAiApi(OpenAiApi.builder()
>                 .baseUrl("https://open.bigmodel.cn/api/paas/v4")  // GLM 的 OpenAI 兼容地址
>                 .apiKey("your-api-key")
>                 .completionsPath("/chat/completions")  // 覆盖默认的 /v1/chat/completions
>                 .build())
>         .defaultOptions(OpenAiChatOptions.builder()
>                 .model("glm-4.7")
>                 .temperature(0.7)
>                 .build())
>         .build();
> ```

## 事件流

开启 `thinkingMode` 后，流式输出中会多出 `Thinking` 事件：

```java
agent.streamForResult("帮我分析这段代码的复杂度", RunnableParams.empty())
        .doOnNext(event -> {
            if (event instanceof AgentStreamEvent.Thinking t) {
                System.out.println("[思考] " + t.content());    // 模型推理过程
            } else if (event instanceof AgentStreamEvent.Text t) {
                System.out.print(t.content());                   // 正式回答
            }
        })
        .blockLast();
```

**开启 thinkingMode**：

```
[AgentStart]
[Thinking] 让我分析一下这段代码...     ← 思考过程（Thinking 事件）
[Text] 这段代码的时间复杂度是...       ← 正式回答（Text 事件）
[Complete]
```

**关闭（默认 DISABLED）**：

```
[AgentStart]
[Text] 这段代码的时间复杂度是...       ← 直接输出正式回答
[Complete]
```

## 关闭思考输出

使用普通模型（如 qwen-plus、glm-4.7）时不需要配置，默认就是 `DISABLED`。

如果使用思考模型但不希望看到思考过程，也保持默认即可：

```java
ReactAgent agent = ReactAgent.builder()
        .chatModel(chatModel)
        // 不配置 thinkingMode，默认 DISABLED
        .build();
```

关闭时框架会自动处理：

- 在用户消息末尾追加 `<no_think>` 标签，告诉模型不输出思考过程
- 如果模型仍然输出了 `<think/>` 内容，会自动剥离，不会出现在最终回答中

## 向后兼容

`thinkTagEnabled(boolean)` 方法仍可用，等价于 `thinkingMode(ThinkingMode.THINK_TAG)`：

```java
// 旧写法（已废弃，但仍可用）
.thinkTagEnabled(true)

// 新写法
.thinkingMode(ThinkingMode.THINK_TAG)
```

## 内部实现

### THINK_TAG 模式

框架通过 `ThinkTagParser` 工具类解析 `<think/>` 标签：

- 跨 chunk 状态追踪：通过 `inThink` 参数跟踪标签开关状态
- 支持 `<think/>` 和 `<think attr="..."/>` 等变体
- 处理跨 chunk 的不完整标签

### REASONING_CONTENT 模式

思考内容已由模型在独立字段中分离，框架直接从 `AssistantMessage.metadata.get("reasoningContent")` 读取：

- 无需字符串解析，直接按字段分发事件
- `reasoning_content` → `Thinking` 事件
- `content` → `Text` 事件
